## 一、项目总体定位
该项目是一个面向机器人操作场景的视觉感知与坐标上报系统。整体目标是：通过相机获取 RGB 与深度图像，使用 YOLO Pose 识别几何目标并估计关键点和姿态角，再结合深度图和相机内参计算目标在相机坐标系下的三维坐标，最后通过手眼外参与机械臂实时末端位姿，将目标转换到机械臂基座坐标系，并进行日志记录或 HTTP 上报。
从工程结构看，项目不是单一脚本，而是由三个子项目共同组成完整闭环：
三者的依赖关系是单向的：标定项目输出外参给 ROS2 项目；训练项目输出模型权重和关键点语义文件给 ROS2 项目；ROS2 项目在运行时读取这些结果并完成实时感知。
## 二、系统总体数据流
在线运行时的数据链路如下：
```
HP60C 相机 RGB 压缩图像
  -> detection_node
  -> /object_centers_2d
  -> depth_node + 深度图 + CameraInfo
  -> /object_centers_3d
  -> base_logger_node + T_cam2ee + WebSocket(link5_hmat)
  -> 基座坐标日志 / HTTP 上报
```
其中 /object_centers_2d 承载目标类别、置信度、二维中心点、姿态角和关键点；/object_centers_3d 在此基础上增加深度值和相机坐标系下的 x、y、z。base_logger_node 再将相机坐标转换到机械臂基座坐标。
## 三、离线流程：手眼标定
handeye-calibration 子项目用于眼在手上配置，即相机安装在机械臂末端附近，需要求出相机坐标系到末端执行器坐标系的变换 T_cam2ee。
### 1. 在线采集
online_capture.py 同时订阅相机图像和机械臂 WebSocket 位姿。图像来自 ROS2 RGB 话题，位姿来自 WebSocket 中的 link5_hmat，该矩阵被视为 T_ee2base。用户按 c 或空格采集时，脚本会检查图像年龄、位姿年龄和二者接收时间偏差，避免保存过期或不同步的数据。
- 输出图像：images/0.jpg、images/1.jpg 等。
- 输出位姿：images/poses.txt，每行格式为 x,y,z,Rx,Ry,Rz，单位为米和弧度。
- 输出元数据：capture_meta.jsonl，包含原始 T_ee2base、时间偏差、WebSocket 序号等，便于质量评估。
### 2. 标定计算
eyeInHand.py 读取标定板图片和 poses.txt，先用棋盘格角点做相机标定，得到相机内参、畸变参数、每张图中标定板相对相机的 rvec/tvec；然后调用 OpenCV calibrateHandEye，分别计算 TSAI、PARK、HORAUD 三种方法的手眼结果。
脚本还会进行质量评估：相机内参重投影误差、多方法旋转/平移一致性、固定靶在基座系下的一致性。如果 capture_meta.jsonl 存在，会优先用固定靶 RMS 选择最优方法；否则回退到方法间平移一致性。
### 3. 自动写回 ROS2 配置
标定结束后，脚本会把最优 T_cam2ee 写入 robot-vision-ros2/config.yaml 的 hand_eye.T_cam2ee。在线运行时的坐标变换链为：p_base = T_ee2base @ T_cam2ee @ p_cam。
## 四、离线流程：YOLO Pose 训练
yolo-pose-pipeline 子项目负责生成适合机器人几何目标的 YOLO Pose 模型。它包含从视频抽帧、LabelMe 标注转换、数据集划分、模型训练和本地实时验证几个步骤。
### 1. 视频抽帧
extract_frames.py 从 data_video 目录批量读取视频，按配置中的 fps 抽帧，默认约每秒 2 帧。每个视频输出到独立子目录，便于后续标注。
### 2. 数据准备
prepare_data.py 支持两种模式：convert 模式将 LabelMe JSON 转为 YOLO Pose txt，并划分 train/val/test；split-only 模式对已有 YOLO txt 只做数据集划分。转换时通过 group_id 将矩形框和关键点归属到同一个目标，并根据 pose_classes.yaml 确定类别顺序。
- 固定关键点数量：当前 max_keypoints 为 4，不足的关键点填 0 并标记不可见。
- 关键点排序：优先按 p1/p2/p3/p4，圆形可优先 center，其余按坐标排序。
- 数据集配置：自动写出 data*.yaml，包含 kpt_shape、flip_idx 和类别 names。
### 3. 模型训练与推理验证
train.py 使用 Ultralytics YOLO，默认从 yolo11n-pose.pt 预训练权重开始训练。训练参数如 epochs、batch、imgsz、run_name、runs_dir 都集中在 config.yaml。realtime_detect_pose.py 可用摄像头直接加载训练好的 best.pt，显示模型输出和 FPS，用于快速验证模型效果。
## 五、在线 ROS2 子系统
### 1. center_depth_msgs
该包定义两个核心消息：Centers2D 和 Centers3D。两者都采用并行数组设计，即 obj_class、score、u、v、angle 等数组长度对应同一批检测目标。Centers3D 在 Centers2D 的基础上增加 depth_m、x、y、z。
### 2. detection_node
detection_node 订阅 RGB 图像，运行 YOLO Pose，发布 /object_centers_2d。其核心不是简单检测框，而是基于关键点计算真实几何中心和姿态角。
- 实时性：使用后台推理线程和 queue(maxsize=1)，只处理最新帧，避免旧帧积压。
- 推理加速：CUDA 下启用 FP16，启动时做 GPU 预热，max_det 限制最坏情况下的 NMS 开销。
- 结果稳定：min_stable_frames 连续稳定帧门控过滤单帧误检，循环 EMA 平滑角度并处理 ±90° 环绕。
- 几何语义：圆形、三角形、梯形、正方形分别定义中心和角度计算规则。
### 3. depth_node
depth_node 使用 ApproximateTimeSynchronizer 同步深度图和 Centers2D，同时订阅 CameraInfo。同步回调只把最新数据放入工作队列，真正的深度解码和反投影在后台线程完成。
- 深度读取：在检测中心周围 sample_radius 区域取有效深度中位数，减少单点噪声影响。
- 深度过滤：小于 min_depth_m 或大于 max_depth_m 的深度被判为无效。
- 反投影公式：x=(u-cx)*z/fx，y=(v-cy)*z/fy，z=depth。
- 无效深度：x、y、z 写 NaN，保留检测目标但标记为不可用三维坐标。
### 4. image_numpy
image_numpy.py 是图像解码工具层，避免依赖 cv_bridge，直接使用 numpy 和 OpenCV。它支持 RGB CompressedImage、原始 Image、compressedDepth PNG/RVL 等格式。RVL 深度解码优先使用首次导入时编译的 C 共享库，失败时回退到 Python 实现。
### 5. object_base_logger
base_logger_node 订阅 /object_centers_3d，通过 WebSocket 读取机械臂 link5_hmat，即 T_ee2base。然后读取 config.yaml 中的 T_cam2ee，计算 p_base = T_ee2base @ T_cam2ee @ p_cam，并输出控制台日志或 POST 到 Head 设备。
- 支持 target_class 过滤，只处理指定类别。
- 支持 log_best_detection_only，只上报置信度最高目标。
- 支持 head_position_z_offset_m，对 HTTP 上报的 z 做额外偏移。
- WebSocket 线程自动重连，并用锁保护最新 T_ee2base。
### 6. visualization_node 与 grasp_node
visualization_node 将 RGB 与最新 Centers3D 解耦显示，不强制同步，因此画面可以按相机帧率刷新，同时叠加最近一次检测结果。grasp_node 则把 Centers3D 转换为抓取 PoseArray 和最优 PoseStamped，包含三维位置 EMA、按 angle_rad 构造偏航四元数、预抓取悬停点等逻辑。

## 七、配置与启动方式
项目大量参数集中在 config.yaml，launch 文件会读取配置作为默认值，命令行 launch 参数优先级更高。robot-vision-ros2/config.yaml 是在线部署的核心配置，包含 hand_eye、camera、model、detection、depth、network、logger 等分区。
常用启动方式：
```
cd robot-vision-ros2
bash scripts/build.sh
bash scripts/run.sh                 # detection + depth + base_logger
bash scripts/run_pipeline_only.sh   # 只启动 detection + depth
```
完整 launch 为 object_base_logger.launch.py，它包含 yolo_center_depth_hp60c.launch.py，后者再包含通用 yolo_center_depth.launch.py。这种分层让通用检测/深度链路与 HP60C 专用话题配置分离。
## 八、工程优化点
- 实时线程模型：检测、深度、可视化、抓取等节点都尽量让 ROS 回调 O(1) 返回，重计算放到后台线程。
- 最新帧策略：图像或同步数据队列 maxsize=1，优先保证低延迟而非处理全部历史帧。
- 压缩传输：RGB 和深度默认使用 CompressedImage/compressedDepth，降低网络和 DDS 带宽压力。
- QoS 选择：传感器数据和检测结果多采用 BEST_EFFORT、KEEP_LAST、depth=1，匹配相机驱动并减少堆积。
- 兼容性处理：不用 cv_bridge，规避 ROS Humble 与 NumPy 2.x 的兼容问题。
- 参数化设计：路径、阈值、话题、网络地址、日志上报都集中配置，便于现场调试。
- 诊断工具：pipeline_doctor.py 可检查 RGB、Depth、CameraInfo、Centers2D、Centers3D、Overlay 的发布者、订阅者和帧数。
## 九、当前系统边界与风险
- 类别级状态：检测稳定计数和角度 EMA 以类别为 key，如果同一类别同时出现多个目标，状态可能相互影响。更稳妥的方案是引入目标跟踪 ID 或按空间邻近匹配。
- 深度/RGB 对齐假设：depth_aligned_to_rgb 默认为 true。如果相机驱动输出未对齐深度，中心像素取深度会产生系统误差。
- 手眼标定质量决定最终 base_xyz 精度。若标定板角点检测质量差、样本姿态覆盖不足或 WebSocket 位姿不同步，最终坐标会偏。
- HTTP 上报是同步 urllib 请求，虽然有较短超时，但在网络异常时仍可能占用 base_logger 回调时间；高频上报场景可改为异步队列。
- 配置和文档中部分历史描述与当前文件名可能不完全一致，例如 CLAUDE.md 提到的旧脚本名与当前 README 中的脚本名不同，维护时应以实际代码为准。

## 十一、结论
本项目整体上是一个较完整的机器人视觉工程闭环：离线阶段完成手眼标定和 YOLO Pose 模型训练；在线阶段通过 ROS2 将 RGB 检测、深度反投影、坐标变换和日志/上报串联起来。技术重点不只是目标检测模型本身，还包括关键点几何后处理、深度有效性处理、手眼坐标链、低延迟 ROS2 通信和现场可配置性。从代码结构看，项目已经围绕实时性做了较多工程优化，后续主要应加强多目标跟踪、测试验证和上报解耦。
